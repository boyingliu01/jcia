# 代码审查清单

## Clean Code

### 命名
- [ ] 命名是否清晰表达意图？
- [ ] 变量名是否避免误导？
- [ ] 函数名是否使用动词/动词短语？
- [ ] 类名是否使用名词/名词短语？
- [ ] 是否避免单字母变量（循环除外）？

### 函数
- [ ] 函数是否只做一件事？
- [ ] 函数是否短小（<20行）？
- [ ] 函数参数是否不超过3个？
- [ ] 是否避免布尔参数？
- [ ] 函数是否有副作用？
- [ ] 是否使用异常而非错误码？

### 注释
- [ ] 注释是否必要且准确？
- [ ] 注释是否解释"为什么"而非"是什么"？
- [ ] 是否删除冗余注释？
- [ ] 是否有误导性注释？

### 格式
- [ ] 代码格式是否一致？
- [ ] 行宽是否不超过120字符？
- [ ] 缩进是否正确？
- [ ] 空行使用是否合理？

### 重复
- [ ] 是否有重复代码（DRY）？
- [ ] 是否可以提取公共函数？

## SOLID

### SRP (单一职责)
- [ ] 类是否有单一职责？
- [ ] 能否用一句话描述类的职责？
- [ ] 修改一个职责是否影响其他职责？

### OCP (开闭原则)
- [ ] 新增功能是否通过扩展实现？
- [ ] 是否需要修改现有代码来添加功能？
- [ ] 是否使用多态/接口？

### LSP (里氏替换)
- [ ] 子类是否可以替换父类？
- [ ] 子类是否违反了父类的契约？
- [ ] 是否有改变行为的地方？

### ISP (接口隔离)
- [ ] 接口是否足够内聚？
- [ ] 客户端是否被迫实现不需要的方法？
- [ ] 接口是否可以拆分？

### DIP (依赖倒置)
- [ ] 是否依赖接口/抽象类？
- [ ] 高层模块是否依赖低层模块？
- [ ] 是否使用依赖注入？

## Clean Architecture

### 依赖规则
- [ ] 依赖方向是否正确（外层→内层）？
- [ ] 领域层是否独立于框架？
- [ ] 是否通过接口解耦？
- [ ] 是否避免依赖穿透？

### 分层结构
- [ ] 每层职责是否清晰？
- [ ] 是否有循环依赖？
- [ ] 接口是否定义正确？

## TDD

### 测试编写
- [ ] 测试是否在实现之前编写？
- [ ] 测试是否描述期望行为？
- [ ] 测试是否独立且可重复？
- [ ] 测试是否快速执行？
- [ ] 是否遵循红→绿→蓝循环？

### 测试覆盖
- [ ] 是否覆盖所有分支？
- [ ] 是否覆盖边界条件？
- [ ] 是否覆盖异常情况？
- [ ] 测试覆盖率是否达标（>80%）？

### 测试质量
- [ ] 测试是否易于理解？
- [ ] 测试命名是否清晰？
- [ ] 是否有脆弱的测试？
- [ ] 是否避免测试实现细节？
