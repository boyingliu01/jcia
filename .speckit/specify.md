# 功能规格

## 背景

JCIA 项目已经实现了基础的 Java 代码变更影响分析功能，但一直不能完整支持**远程调用**的分析。这是一个核心痛点，因为现代 Java 应用大量使用 RPC（Dubbo、Feign）、HTTP API、消息队列等方式进行服务间通信，无法分析这些调用链会导致影响范围评估不完整。

用户希望用严格的工程化方法从头再来一遍，确保：
1. 遵循 SDD/TDD/Clean Architecture 原则
2. 完整支持远程调用分析
3. 代码质量和测试覆盖率达标

## 目标

### 主要目标

1. **完善远程调用分析能力**：支持 Dubbo、Feign、HTTP Client、消息队列等调用方式
2. **重构核心模块**：按照严格工程化标准重构现有代码
3. **提升测试覆盖率**：达到 80% 以上整体覆盖率

### 预期结果

- 完整的调用链分析（本地 + 远程）
- 准确的影响范围评估
- 高质量的代码库

## 功能需求

### 需求1：远程调用检测

- 描述：检测 Java 代码中的远程调用模式
- 调用类型：
  - Dubbo RPC（@DubboReference、@Reference）
  - Spring Cloud Feign（@FeignClient）
  - HTTP Client（RestTemplate、OkHttp、HttpClient）
  - 消息队列（@RabbitListener、@KafkaListener、RocketMQ）
- 接受标准：
  - 正确识别至少 90% 的远程调用点
  - 提供调用端点信息（服务名、方法名、URL）
- 拒绝标准：
  - 遗漏常见远程调用模式
  - 误报率 > 10%

### 需求2：远程调用链追踪

- 描述：追踪远程调用的影响传播
- 能力要求：
  - 记录调用关系（调用方 → 远程端点）
  - 支持跨服务影响标记
  - 生成完整的调用拓扑图
- 接受标准：
  - 影响链包含远程调用节点
  - 正确标注跨服务边界
- 拒绝标准：
  - 远程调用被当作普通方法处理
  - 缺失跨服务影响标记

### 需求3：分析结果融合

- 描述：融合多分析源的结果
- 分析源：
  - 静态分析（源代码解析）
  - CodeQL 语义分析
  - 运行时追踪（SkyWalking）
  - 远程调用分析
- 接受标准：
  - 结果不冲突、不遗漏
  - 提供置信度评分
- 拒绝标准：
  - 结果冲突未处理
  - 重复或遗漏节点

### 需求4：严重度评级增强

- 描述：基于远程调用影响调整严重度评级
- 评级因素：
  - 调用链深度
  - 跨服务数量
  - 业务重要性（核心服务 vs 边缘服务）
- 接受标准：
  - 远程调用影响有独立评级维度
  - 高风险调用有明确标记
- 拒绝标准：
  - 严重度评级未考虑远程调用因素

## 非功能需求

### 性能要求

- 大型项目（10万行代码）分析时间 < 5分钟
- 内存占用 < 2GB

### 安全要求

- 不暴露敏感信息（API密钥等）
- 输出报告不含敏感数据

### 可用性要求

- CLI 命令简洁直观
- 输出报告易于理解
- 错误信息清晰有指导性

### 兼容性要求

- Python 3.10+
- Windows/Linux/macOS 跨平台
- Git 仓库支持

## 限制条件

1. 远程调用分析依赖源代码可见性（无法分析闭源服务）
2. 运行时追踪需要 SkyWalking 等工具部署
3. CodeQL 需要 GitHub Enterprise 或本地安装
4. 消息队列分析需要消息元数据
