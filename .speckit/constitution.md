## 项目概述

项目名称：JCIA (Java Code Impact Analyzer)

JCIA 是一个用于分析 Java 项目中代码变更影响范围的工具。它识别受影响的类和方法，智能选择测试案例，并执行回归测试。采用 Clean Architecture（六边形架构）原则实现。

### 项目使命

为 Java 开发者提供准确、高效的代码变更影响分析能力，帮助团队：
1. 精准识别变更影响范围
2. 智能选择测试用例
3. 支持现代 Java 生态（RPC、HTTP、消息队列）
4. 提供可衡量的开发价值

## 核心原则

### 1. 开发方法论

- **SDD (Specification-Driven Development)**: 先规格，后实现
  - 所有功能必须在 `.speckit/specify.md` 中定义规格
  - 规格必须包含可测试的验收标准
  - 规格变更需评审并更新文档

- **TDD (Test-Driven Development)**: 先测试，后代码
  - 任何新功能必须先有失败的测试
  - 代码实现以通过测试为目标
  - 重构在测试保护下进行

- **Clean Architecture**: 六边形分层架构，依赖规则
  - 依赖必须指向内层
  - 外层通过接口依赖内层
  - 禁止跨层直接依赖

- **Clean Code**: 可读性、可维护性优先
  - 代码是自文档化的
  - 注释解释"为什么"而非"是什么"
  - 函数短小，单一职责

- **SOLID**: 面向对象设计原则
  - Single Responsibility: 类只有一个变化理由
  - Open/Closed: 对扩展开放，对修改关闭
  - Liskov Substitution: 子类可替换父类
  - Interface Segregation: 接口小而专注
  - Dependency Inversion: 依赖抽象而非具体

### 2. 代码规范

- **命名清晰表达意图**（camelCase/snake_case）
  - 函数名：动词或动词短语（`calculate_impact`）
  - 类名：名词或名词短语（`ImpactAnalyzer`）
  - 常量：全大写下划线（`MAX_DEPTH`）
  - 布尔变量：肯定语气（`is_valid` 而非 `is_not_valid`）

- **函数单一职责，短小精悍**
  - 函数长度 < 20 行（理想）
  - 参数数量 < 4 个
  - 嵌套深度 < 3 层
  - 只做一件事，做好一件事

- **避免重复代码 (DRY)**
  - 提取公共逻辑到函数/类
  - 使用继承或组合复用代码
  - 单一事实来源（Single Source of Truth）

- **注释解释"为什么"而非"是什么"**
  - 好的代码是自解释的
  - 注释说明设计决策、业务规则、复杂算法
  - TODO 注释必须有追踪编号或日期

- **遵循 Python 最佳实践（PEP 8 + Ruff 规则）**
  - 使用 Python 3.10+ 语法（`str | None` 而非 `Optional[str]`）
  - 类型注解覆盖所有公共 API
  - 导入排序：标准库 > 第三方 > 本地

### 3. 工程要求

- **所有提交必须通过 Ruff Lint 检查**
  - 提交前运行 `make lint`
  - 零警告容忍（包括文档字符串、类型注解）
  - 使用 `make format` 自动格式化

- **所有提交必须有测试覆盖**
  - 新代码必须有对应测试
  - 覆盖率不能下降（维持 ≥ 80%）
  - 关键路径必须有 100% 分支覆盖

- **新功能必须先写测试（TDD）**
  - Red-Green-Refactor 循环
  - 测试即规格，测试即文档
  - 提交历史清晰展示 TDD 过程

- **代码审查必须通过检查清单**
  - 功能正确性
  - 测试充分性
  - 架构合规性
  - 代码清晰性

- **童子军规则：每次提交改善代码质量**
  - 看到脏乱就清理（即使不是你写的）
  - 渐进式改进优于完美主义拖延
  - 提交信息解释"为什么"和"做了什么"

## 架构原则

### 分层结构

```
Entities（核心领域）→ Interfaces（抽象契约）→ Services（领域逻辑）→ Use Cases（应用编排）→ Adapters（外部集成）
```

### 依赖规则

- **依赖必须指向内层**
  - 外层知道内层，内层不知道外层
  - 通过接口/抽象依赖，而非具体实现

- **Entities 不依赖任何外部模块**
  - 纯数据和行为，无框架依赖
  - 可独立测试，可复用

- **Services 只依赖 Entities**
  - 领域逻辑协调
  - 无外部系统知识

- **Use Cases 依赖接口，不直接依赖适配器**
  - 应用流程编排
  - 通过端口（接口）与外部交互

- **Adapters 作为外部系统的桥梁**
  - 实现接口
  - 处理技术细节（HTTP、数据库、文件等）
