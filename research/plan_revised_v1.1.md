# JCIA 项目开发计划（修订版）

## 文档信息
- **版本**: 1.1.0（修订版）
- **修订日期**: 2026-02-01
- **修订依据**: 基于国内外大厂（Google、阿里、腾讯、美团、字节、百度、华为、京东）精准测试实践研究报告
- **当前进度**: Phase 4/13 已完成（Domain Entities & Core Interfaces）

## 修订概述

### 为什么需要修订计划？

通过深入研究国内外大厂（Google、阿里、腾讯、美团、字节、百度、华为、京东等）的精准测试实践，发现以下关键洞察：

1. **工程化 > 算法先进性** - 美团AlphaTest成功在于"好用"而非算法先进
2. **可视化与可解释性至关重要** - 百度、字节强调"可追溯、可量化、透明化"
3. **渐进式落地是最佳策略** - 华为、百度都强调分阶段演进
4. **CI/CD集成是核心竞争力** - 所有大厂都将精准测试作为研发流程的一部分

### 修订核心方向

| 维度 | 原规划 | 修订后 |
|------|--------|--------|
| **MVP范围** | 完整5阶段工作流 | 聚焦变更分析+测试选择 |
| **算法策略** | STARTS算法为主 | 静态依赖+动态调用链混合 |
| **报告重点** | 多种格式报告 | 可视化影响图+推荐理由 |
| **集成方式** | 后期考虑 | MVP阶段即考虑CLI+CI集成 |
| **AI功能** | 测试生成（Phase 4） | 延后至Phase 2，先保证核心 |

### Phase 5: 变更影响分析引擎（当前Phase - 优先级保持）

**目标**: 实现代码变更的静态影响分析能力

**核心任务**（基于大厂实践调整）：

1. **Java代码静态分析**
   - [ ] 使用javalang解析Java源码AST
   - [ ] 提取类依赖关系（extends/implements/field类型）
   - [ ] 提取方法调用关系（方法内部调用其他方法）
   - [x] 变更文件和方法识别（Git diff）

2. **调用链图构建** 
   - [ ] 基于AST构建类级别依赖图
   - [ ] 基于AST构建方法级别调用图
   - [ ] 支持向上追溯（谁调用了我）和向下追溯（我调用了谁）
   - [ ] 图数据结构设计（适配networkx）

3. **影响范围计算**
   - [ ] 直接变更标记
   - [ ] 一级依赖识别（直接上下游）
   - [ ] 多级依赖传播算法
   - [ ] 影响严重程度评估（基于调用深度和频次）

**调整说明**:
- **增加**: 可视化影响图（参考字节、美团的可视化实践）
- **调整**: 优先保证类级别分析准确，方法级别可以后续细化
- **参考**: 华为MFQ方法中的影响分析理念

**交付物**:
- `jcia analyze` 命令可生成影响分析报告
- 支持输出JSON/Console两种格式
- 可识别变更的上下游依赖

### Phase 6: 测试选择算法（优先级提升）

**目标**: 实现基于变更的测试用例智能选择

**核心任务**（结合STARTS+大厂实践）：

1. **测试用例发现**
   - [ ] Maven Surefire测试扫描
   - [ ] 测试类与生产代码关联分析
   - [ ] 基于命名约定的快速匹配（*Test.java）

2. **测试-代码关联** 
   - [ ] 静态关联：测试类import/使用生产类分析
   - [ ] 动态关联（可选Phase 7）：覆盖率数据采集
   - [ ] 关联关系存储（SQLite）

3. **测试选择算法**
   - [ ] **STARTS算法实现**（静态类级别）
     - 基于类依赖图选择测试
     - 支持传递闭包计算
   - [ ] **基于影响的选择**
     - 直接影响的测试（高优先级）
     - 间接影响的测试（中优先级）
     - 可能影响的测试（低优先级）
   - [ ] **排序策略**
     - 默认：影响深度排序
     - 可选：历史失败率排序（预留接口）

4. **推荐理由生成**
   - [ ] 为每个选择的测试生成理由
   - [ ] 显示关联的变更文件/方法
   - [ ] 显示影响路径（A→B→C）

**调整说明**:
- **关键新增**: 推荐理由生成（字节、百度都强调可解释性）
- **简化**: 先实现静态选择，动态覆盖率可以延后
- **参考**: 字节跳动的打分排序机制（简化版）

**交付物**:
- `jcia select` 命令可选择测试用例
- 输出优先级排序的测试列表
- 包含选择理由和预估影响

## MVP（最小可行产品）定义

### MVP边界（Phase 5-8）

**包含功能**:
1. ✅ Git变更分析（文件+方法级别）
2. ✅ 类级别静态依赖分析
3. ✅ 调用链图构建（类级别）
4. ✅ 影响范围计算（上下游识别）
5. ✅ 测试用例静态发现（Maven项目）
6. ✅ STARTS算法测试选择
7. ✅ 测试推荐理由生成
8. ✅ CLI工具完整命令链
9. ✅ 控制台+HTML报告
10. ✅ CI/CD集成（GitHub Actions/Jenkins）

**不包含功能**（后续Phase实现）:
- ❌ 方法级别调用链（太细粒度，先做类级别）
- ❌ JaCoCo覆盖率集成（延后到Phase 9）
- ❌ 历史数据智能推荐（延后到Phase 11）
- ❌ AI测试生成（延后到Phase 13）
- ❌ 回归分析（延后到Phase 10）

### MVP价值验证

基于大厂实践，MVP应实现的价值：

| 价值维度 | MVP目标 | 大厂参考 |
|---------|--------|---------|
| 测试执行时间 | 减少20-30% | 字节降低60%（MVP保守） |
| 人效提升 | 减少10-20%人工评估 | 字节提升30% |
| 分析准确性 | 影响范围识别率>80% | 美团成功率>94% |
| 易用性 | CLI<5分钟上手 | 美团AlphaTest零门槛 |

2. 字节跳动、美团一期都没有AI生成，二期才考虑
3. 核心功能（分析+选择）没有AI也能产生价值
4. 避免过度工程，先解决核心痛点

### 决策2：为什么把CI/CD集成从后期提前到Phase 8？

**原因**:
1. Google TAP、美团、字节都将CI集成作为核心场景
2. 没有CI集成的测试工具难以在实际项目中落地
3. 参考pytest-testmon的CLI-first策略
4. CI集成相对独立，不阻塞其他功能开发

### 决策3：为什么先实现类级别而非方法级别分析？

**原因**:
1. STARTS算法就是类级别的，已验证可行
2. 类级别分析速度更快，适合大型项目
3. 对于大多数场景，类级别精度已足够
4. 方法级别可以作为Phase 12的性能优化
